The system recommended selling a position that no longer existed

On Apr 17 2026 the daily build emitted a SELL recommendation for NVDY with the rationale "keep a portion for income." One problem: the entire NVDY position had been closed on Mar 27 — a full, profitable exit, recorded in the trade journal. The recommendation was for a holding that did not exist.

How it happened

  • The build's position-state reader used a cached snapshot from an earlier pipeline stage, not the canonical holdings file.
  • Nothing in the pipeline cross-checked action rows against current holdings, so the phantom SELL sailed through rendering and logging.
  • It sat in the outcome journal as RECOMMENDED for nine days before a manual review caught it.

Why this matters more than one bad row

A recommendation for a nonexistent position is a state-integrity failure, not a judgment failure. If the system can be wrong about what it holds, every downstream number — exposure, concentration, risk — inherits that error silently. This is the class of bug that erodes trust fastest, because the output still looks plausible.

Governance fix: the holdings file became the single source of truth for "what is currently held," enforced by a mechanical reconciliation pass on every build. Error vocabulary: ZOMBIE-SELL and ZOMBIE-HOLD (action rows naming unheld tickers, hard errors), PHANTOM-ADD (ADD verb on a fresh ticker), ORPHAN-HOLDING (a real position the output forgot), QTY-MISMATCH (stale share counts).

What the ledger shows

The original zombie receipt stays in the record — that is the point. A verifier can see the bad row, the nine-day gap, and the date the reconciliation gate landed. After the gate, a recommendation naming an unheld ticker cannot ship at all.

Selling governance means showing the bug, the blast radius, and the mechanical fix — not pretending the bug never happened.
