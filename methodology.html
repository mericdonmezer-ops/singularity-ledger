Methodology

How claims enter the chain, how outcomes are scored, and what integrity guarantees exist.

Two tiers

  • LIVE — first published while the outcome was still unknown. This is the verifiable tier.
  • ARCHIVE — backfilled from internal logs after outcomes were knowable. Real history, not pre-registration.

Data flow

  1. Run-engine decisions land in append-only warehouse streams (action-outcomes, rejected-outcomes, forecast-outcomes, etc.).
  2. packages/ledger extracts, scrubs sensitive fields, and merges into ledger.json.
  3. Each build appends a hash-chained block — editing a published claim breaks npm run verify.
  4. Static site renders aggregates: Brier score, hit rate, honesty metrics (correct avoids vs missed winners).

Scoring

  • Forecasts — Brier score on resolved probability calls. Below 0.18 = good; above 0.25 = not earned yet.
  • Rejections — forward return on names we said no to (20d window).
  • Exits — post-sell drift vs thesis (good exit vs churn).

Publish QC (Faz 0+)

Build fails if public text contains mojibake, unfilled placeholders ([n], TBD), or empty claims. Position share counts are stripped — only rationale and policy thresholds remain.

Integrity

Claim hash covers immutable fields (id, type, date, ticker, statement, probability, resolve-by). Resolution fields may update; claims may not. Git history of the public repo is a second tamper-evidence layer.
